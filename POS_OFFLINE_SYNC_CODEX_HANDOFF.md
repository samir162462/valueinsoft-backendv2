# POS Offline Sync Module — Codex Handoff

## Project Context

- **Project**: ValueINSoft SaaS POS System
- **Backend**: Spring Boot 3.4.5, Java 21, JdbcTemplate (NO JPA/Hibernate)
- **Database**: PostgreSQL with per-tenant schemas (`c_{companyId}`)
- **Frontend**: React JS (web), React Native (future mobile)
- **Multi-tenancy**: Each company has its own PostgreSQL schema, each tenant has branches, POS is branch-based
- **Build**: Maven, compiles successfully with all baseline code in place
- **Root package**: `com.example.valueinsoftbackend`
- **Offline sync package**: `com.example.valueinsoftbackend.pos.offline`

## PART A — WHAT IS ALREADY IMPLEMENTED (BASELINE)

### A1. Flyway Migrations (3 files)

**`src/main/resources/db/migration/V74__create_pos_offline_sync_tables.sql`**
- Creates 8 tables in PUBLIC schema (not tenant schema):
  1. `pos_device` — Registered POS devices per tenant/branch
  2. `pos_sync_batch` — One sync attempt from one POS device
  3. `pos_offline_order_import` — Raw offline order payloads + processing status
  4. `pos_idempotency_key` — Prevents duplicate order creation
  5. `pos_offline_order_error` — Detailed validation/processing errors
  6. `pos_bootstrap_version` — Offline cache data versioning per company/branch/data_type
  7. `pos_device_session` — Device/cashier session tracking
  8. `pos_sync_audit_log` — Audit trail for all sync events
- All tables use: BIGINT GENERATED BY DEFAULT AS IDENTITY PKs, TIMESTAMPTZ, JSONB, CHECK constraints, FKs
- All tables have `company_id` and `branch_id` columns for tenant scoping

**`src/main/resources/db/migration/V75__create_pos_offline_sync_indexes.sql`**
- Targeted indexes on all 8 tables
- Uses partial indexes (`WHERE ... IS NOT NULL`) where appropriate

**`src/main/resources/db/migration/V76__alter_sales_tables_for_offline_sync.sql`**
- Creates PL/pgSQL function `public.add_offline_sync_columns_to_table(schema, table)` that safely adds columns:
  `source_channel`, `sync_batch_id`, `device_id`, `offline_order_no`, `idempotency_key`, `client_type`, `local_created_at`, `synced_at`
- Cannot ALTER dynamic tenant tables (e.g. `"PosOrder_1"`) in plain SQL — function must be called programmatically per tenant

10. After all orders: call `batchRepo.updateSummary(...)` with counts of synced/failed/duplicate/needsReview

### TODO 3: Full Order Validation

**File**: `OfflineOrderValidationService.java` (method `validateOrder`, around line 42)

**Task**: Implement business rule validation for each offline order. Currently only checks structural fields.

**What to implement**:
1. **Product validation**: For each item, check `productId` exists in the tenant's `inventory_product` table (use `TenantSqlIdentifiers.inventoryProductTable(companyId)`). Check product is active.
2. **Price validation**: Compare `unitPrice` in the order against the current price in the database. Decide on acceptable threshold (exact match? ±X%?). Use error code `PRICE_CHANGED`.
3. **Tax validation**: Verify `taxAmount` and `taxRate` calculations are mathematically correct.
4. **Discount validation**: Verify any discounts are within allowed rules.
5. **Payment total validation**: Sum all `OfflinePaymentRequest.amount` values and verify they equal `totalAmount`. Use error code `PAYMENT_TOTAL_MISMATCH`.
6. **Stock validation**: Check `inventory_branch_stock_balance` table has sufficient quantity. Use error code `STOCK_UNAVAILABLE`.
7. **Shift validation**: If `localShiftId` is provided, verify a matching open shift exists in `"PosShiftPeriod"` table. Use error code `SHIFT_CLOSED`.
8. **Cashier validation**: Verify `cashierId` exists and is active for the branch.
9. **Branch validation**: Verify `branchId` is active and belongs to `companyId`.
10. **Offline window validation**: Compare `offlineStartedAt` against `device.maxOfflineHours`. Use error code `OFFLINE_WINDOW_EXCEEDED`.

**For each validation failure**: Call `SyncErrorService.saveError(...)` with the appropriate `OfflineSyncErrorCode`, `OfflineErrorSeverity`, field path, and raw value.
